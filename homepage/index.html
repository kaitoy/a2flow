The concept

ITSM workflows, reimagined

Classic workflow engines encode a process as rigid forms, ticket queues, and handoffs between people. Most of the elapsed time is waiting. A2Flow keeps the two things that matter — the procedure and the approvals — and hands everything in between to an agent.

The ticket pipeline

Process as software: fixed forms, fixed states, humans as glue.

  • Fill in the formOne field wrong and it bounces back.
  • Wait in the queueYour request is #14 in line.
  • Hand off between teamsContext is lost at every hop.
  • Chase the status“Any update on my ticket?”

The agent workflow

Process as conversation: the agent drives, humans decide.

  • Describe the intent in chatNo form. The Skill knows what to ask.
  • The agent plans a task graphConcrete steps with dependencies — visible to everyone.
  • Humans approve the moments that matterThe plan itself, and every destructive step.
  • The agent executes and verifiesLive progress in the same conversation.
The procedure itself is not code locked inside a workflow engine. It is an Agent Skill — a SKILL.md file in a Git repository — versioned, reviewable, and readable by anyone on the team.
How it works

Plan. Approve. Execute.

Every workflow session runs under the same two-phase policy: nothing executes until a human has seen the plan, and nothing destructive executes until the designated approver has said yes.

STEP 1 — PLAN

The agent turns intent into a task graph

It reads the linked Agent Skill, breaks the request into WorkflowTasks with explicit dependencies, and registers the whole DAG in one call — cycle detection included. Nothing runs yet.

STEP 2 — APPROVE

Humans gate the moments that matter

The session owner approves the plan before execution starts. Destructive steps gate again on a designated approver: Approve / Reject controls render right in the chat, and only that person can resolve them — anyone else gets a 403.

STEP 3 — EXECUTE

The agent walks the graph

Tasks move pending → in_progress → completed in dependency order, calling only the MCP tools bound to the current task. Everyone watches live, in a table or as a graph.

Built for teams

Three roles, one conversation

A workflow session is a shared chat. The applicant, the approver, and the agent all post into the same thread — no side channels, no “any update on my ticket?”

M

The applicant

Owns the session. Describes the intent, answers the agent's questions, and approves the plan before anything runs.

S

The approver

Gets a notification, opens the session, and decides right in the chat — with a comment. Only the designated approver can resolve an approval.

✦

The agent

Plans the task graph, requests approvals at the right moments, executes each step per the Skill, and reports progress as it goes.

Notifications that carry you to the decision

Approval requests and session completions land in each user's notification bell and deep-link straight into the relevant conversation.

An audit trail on every approval

Who approved, what they wrote, and exactly when — persisted on the approval record and browsable in the admin console.

Features

Everything a workflow needs, nothing it doesn't

Agent Skills in Git

Procedures live as SKILL.md files in Git repositories, cloned on first run. Version them, review them, read them — like any other code.

A live task graph

Tasks form a real DAG — dependencies enforced, cycles rejected — and render as a sortable table or an auto-laid-out graph, updating as the agent works.

Approval gates mid-run

Before a destructive step, the agent requests approval from a specific user and pauses. Approve / Reject buttons appear in the chat — for that user only.

MCP tools, scoped per task

Register remote MCP servers, and the agent binds only the tools each task needs at plan time. At run time, a task can call nothing but its own tools.

Rich UI inside the chat

Through the A2UI protocol the agent renders buttons, forms, and cards inline — and your clicks flow back into the run as tool results.

Bring your own LLM

Gemini by default; OpenAI and Anthropic models via LiteLLM. Swapping providers is one environment variable.

Under the hood

An open-protocol stack

A Next.js chat UI streams AG-UI events from a FastAPI backend that hosts a Google ADK agent. Skills come from Git, tools come from MCP servers, and one relational database holds it all.

Get started

Run it on your machine

The whole stack — PostgreSQL, backend, frontend — comes up with Docker Compose. Bring a Google API key and you're two commands away from your first workflow. 

$ git clone https://github.com/kaitoy/a2flow.git && cd a2flow
$ echo GOOGLE_API_KEY=your_google_api_key_here > .env
$ docker compose up --build
# open http://localhost:3000
